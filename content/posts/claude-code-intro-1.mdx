## 시작하기

Claude Code는 터미널에서 바로 쓰는 에이전트형 코딩 도구입니다. 이 글에서는 설치부터 첫 세션 실행까지를 차근차근 따라가 봅니다. 설치 환경은 macOS와 Windows(WSL)를 기준으로 합니다.

작업의 정본은 언제나 문서에 있습니다. 도구는 그 규칙을 빠르게 실행해 줄 뿐, 규칙 자체를 대신 정하지 않습니다. 그래서 처음 몇 분은 설치와 환경 확인에 쓰고, 그다음부터 실제 작업 흐름을 익히는 편이 좋습니다.

```bash
# 전역 설치 후 실행
pnpm add -g @anthropic-ai/claude-code
claude
```

인라인 수식 $E = mc^2$ 과 블록 수식도 문제없이 렌더됩니다. 수식이 많은 글에서도 목차는 헤딩만 추려 보여주므로 본문 흐름을 해치지 않습니다.

$$
\int_0^1 x^2 \, dx = \frac{1}{3}
$$

설치에 앞서 알아둘 점은, Claude Code가 프로젝트 폴더 단위로 동작한다는 것입니다. 작업할 저장소 안에서 실행하면 그 폴더의 파일들을 맥락으로 삼아 움직입니다. 그래서 설치 직후에는 아무 프로젝트에서나 한 번 실행해 보며 감을 잡는 것이 좋습니다.

## 설치

설치는 패키지 매니저로 진행합니다. 이 프로젝트는 pnpm을 전용으로 쓰므로 예제도 pnpm 기준입니다. npm이나 yarn을 섞으면 잠금 파일이 충돌하니 하나로 통일하는 편이 안전합니다.

설치 자체는 명령 한 줄이면 끝나지만, 실제로는 셸 설정과 PATH를 한 번 점검하는 게 좋습니다. 전역 설치한 실행 파일이 PATH에 잡혀 있어야 어느 폴더에서든 `claude`를 바로 부를 수 있기 때문입니다. 설치 후에는 버전 확인으로 정상 여부를 검증합니다.

아래 두 환경을 각각 다룹니다. 운영체제마다 셸과 권한 처리가 조금씩 달라, 처음 한 번만 짚어두면 이후에는 신경 쓸 일이 거의 없습니다.

### macOS

macOS에서는 Homebrew나 pnpm 전역 설치를 사용할 수 있습니다. 여기서는 pnpm 전역 설치를 권장합니다. 터미널을 열고 설치 명령을 실행하면 몇 초 안에 끝납니다. 설치가 끝나면 `claude --version`으로 버전을 확인하세요. 버전 문자열이 보이면 정상입니다.

zsh를 기본 셸로 쓰는 경우 `~/.zshrc`에 pnpm 전역 bin 경로가 들어가 있는지 확인합니다. 대부분 pnpm 설치 과정에서 자동으로 추가되지만, 수동으로 관리하는 환경이라면 한 번 살펴보는 게 좋습니다. 경로가 빠져 있으면 셸을 새로 열었을 때 명령을 못 찾을 수 있습니다.

권한 관련해서는 별도 설정이 거의 필요 없습니다. 다만 회사 기기처럼 보안 정책이 강한 환경에서는 실행 파일 서명 검사에 걸릴 수 있으니, 그럴 때는 관리자에게 예외 처리를 요청하면 됩니다. 여기까지 오면 macOS 설치는 마무리됩니다.

### Windows

Windows에서는 WSL 환경을 강하게 권장합니다. 네이티브 PowerShell에서도 동작하지만, 경로 구분자와 권한 모델이 유닉스 계열과 달라 자잘한 마찰이 생기기 쉽습니다. WSL을 쓰면 리눅스와 동일한 경험을 얻을 수 있어 문서의 예제를 그대로 따라 하기 편합니다.

WSL 배포판(예: Ubuntu)을 설치한 뒤에는 macOS와 동일하게 pnpm 전역 설치를 진행하면 됩니다. 배포판 안에서 셸을 열고 같은 명령을 실행하면 되고, 버전 확인 방법도 같습니다. 파일은 가급적 WSL 파일시스템 안(`~` 아래)에 두는 편이 입출력 성능에 유리합니다.

네이티브 환경을 꼭 써야 한다면 Windows Terminal과 함께 사용하는 것을 권합니다. 여기까지 오면 설치 단계는 모두 끝났고, 이제 첫 실행으로 넘어갈 준비가 되었습니다.

## 첫 실행

설치가 끝났으면 프로젝트 폴더로 이동해 `claude`를 실행합니다. 첫 실행에서는 로그인과 기본 권한 설정을 안내받습니다. 안내에 따라 인증을 마치면 대화형 세션이 열립니다.

세션이 열리면 자연어로 작업을 지시할 수 있습니다. 예를 들어 "이 폴더 구조를 요약해 줘"라고 하면 파일을 훑어 정리해 주고, "테스트를 실행하고 실패를 고쳐 줘"처럼 여러 단계를 한 번에 맡길 수도 있습니다. 처음에는 작은 작업부터 시켜 보며 도구의 반응을 익히는 것을 추천합니다.

권한은 필요할 때마다 물어보도록 설정하는 편이 안전합니다. 특히 커밋이나 원격 반영처럼 되돌리기 어려운 작업은 항상 확인을 거치도록 두는 것이 좋습니다. 익숙해지면 이 확인 단계를 신뢰하는 범위 안에서 조금씩 줄여 갈 수 있습니다.

### 세션 종료

작업이 끝나면 세션을 종료합니다. 종료해도 대화 맥락은 요약되어 다음 세션에서 자연스럽게 이어집니다. 그래서 긴 작업을 여러 세션에 나눠 진행해도 맥락이 크게 끊기지 않습니다.

종료 전에는 변경 사항을 한 번 훑어보는 습관을 들이면 좋습니다. 무엇이 바뀌었는지, 커밋할 준비가 되었는지 스스로 점검하는 것이죠. 이 짧은 점검이 다음 세션의 출발을 훨씬 가볍게 만들어 줍니다.
