> 각 페이지의 역할·구성·정책을 확정한다. 데이터 계약은 `content-plan.md`, 스타일 토큰은 `design.md`, 스택 근거는 `tech-stack.md`.
> 원칙: **"글에 집중"** — 산만 요소는 빼고, 읽기를 돕는 것만.

## 0. 페이지 지도

nav = **Home · Posts · Series · About** (+ RSS, 다크 토글)

| 페이지 | 경로 | 역할 | 렌더링 |
|---|---|---|---|
| Home | `/` | 대표 글 진입점 ("가장 많이 본") | **ISR** (조회수 반영) |

## 1. Home (`/`) — 가장 많이 본 + 최근 글

### 구성
- **Masthead** (design.md §4.1: brand + lede + nav)
- **"가장 많이 본" 섹션** — 조회수 상위 N개(기본 **N=5**). 줄기-잎 목록 재사용.
- **"최근 글" 섹션** — 최신 날짜 N개(기본 **N=5**). 빌드타임 Velite 데이터(정적).
- 하단: "전체 글 보기 →" (`/posts`로)

> 두 섹션은 **독립**(겹침 허용 — 확정 2026-07-01) — "인기"와 "신규"는 다른 질문에 답하므로 **중복 노출을 허용한다.** 최근 글에서 인기글을 따로 제외하지 않는다.

### 렌더링 — ISR (긴 주기)
- **`revalidate = 86400`(하루)** 추천. 근거:
  - **새 글**은 push→배포 시 전체 재빌드로 즉시 반영 → "최근 글"은 ISR 타이머와 무관하게 항상 최신.
  - ISR 타이머가 필요한 **유일한 이유는 조회수 랭킹 신선도**. 주 1편 블로그에서 인기 순위는 천천히 바뀌므로 시간당 갱신은 과함.
  - 하루 1회면 충분(읽기 월 ~30회 → 무료 한도 대비 무시 가능). 더 느긋하게 원하면 주기를 늘려도 됨.

### 데이터
- 조회수 랭킹은 `entities/post`의 셀렉터가 Redis 정렬셋 top-N을 읽고, **항상 Velite 현재 글과 교집합**한 뒤 메타를 붙인다.
- "최근 글"은 순수 Velite(날짜 정렬)라 Redis와 무관 — Home의 정적 뼈대. (조회수 시스템 상세·폴백 → §5)

> **ISR 동작·제한 (참고):** App Router에서 `export const revalidate = 86400`만 두면 됨. 빌드 시 정적 생성 → 방문자는 CDN 캐시를 즉시 받고, 24h 만료 후 첫 요청에 **낡은 버전을 먼저 주고 백그라운드에서 재생성**(stale-while-revalidate) → 아무도 로딩을 안 기다림. 재생성 실패 시 **직전 성공본 유지**(§5.2 폴백과 맞물림). **Vercel Hobby 무료 한도**: ISR Writes 20만/월(우리 ≈30), ISR Reads 100만/월, 함수호출 100만/월 → 모두 1% 미만. 초과 시 과금이 아니라 **기능 30일 일시정지**. 단 Hobby는 **비상업·개인용** 전용(광고·유료 스폰서 시 Pro 필요).

## 2. Posts (`/posts`, `/posts/page/[n]`) — 전체 글, 날짜순

### 구성
- 글 목록 = design.md §4.2 (줄기-잎 + 커버 썸네일 변형)
- **날짜 내림차순**(최신 우선), `draft:true` 제외
- 하단 **페이지네이션 컴포넌트**

### 정렬 규칙 (확정 2026-07-02 — `test-plan.md §3.1`)
- **복합 정렬 `date desc → slug asc`**: 날짜 내림차순이 1차, **같은 날짜면 slug 사전순**이 2차. 날짜만으로 정렬하면 동점 순서가 빌드마다 흔들려 ISR·OG 스냅샷이 불안정해지므로 **결정적 tie-break**를 못박는다.
- **미래 날짜 글은 그냥 노출**: 예약 발행 개념이 없으므로 `date`가 미래여도 필터링하지 않고 정렬만 한다. (추후 예약 발행이 필요해지면 이 정책부터 바꾼다)
- 이 규칙은 `getPublishedPosts()` 셀렉터 한 곳에서 보장하며, 목록·RSS·시리즈·페이지네이션 전부의 상류다.

### 페이지네이션 정책
- **경로 기반 SSG**: 1페이지는 `/posts`(정규 URL), 2페이지부터 `/posts/page/2` … `generateStaticParams`로 전부 프리렌더.
- **페이지 크기: 10** (추천 기본값 — 조정 가능)
- 컴포넌트: 미니멀 숫자 페이저 `‹ 1 2 3 … M ›`. 현재 페이지 강조(ink+moss), 나머지 stone. 사이드 요소 아님(하단 고정).
- 접근성: `<nav aria-label="페이지네이션">`, 현재 페이지 `aria-current="page"`.

## 3. Series (`/series` → `/series/[slug]`)

### 3.1 Series 목록 (`/series`) — 진행/완결 그룹 카드 그리드
- **두 그룹으로 구분:** **"진행 중"** 섹션 먼저, **"완결"** 섹션 아래. 시리즈 개수와 무관하게 항상 이 구조. **페이지네이션·더보기 없음** — 시리즈는 본질적으로 적으므로(수년간 수십 개) 그룹핑이 스케일 정책 그 자체.
- **완결 판정 = series yml 명시 플래그 `complete: true`** (기본 `false` = 진행 중). 날짜 비활동 추론 안 함 — `order` 필수 결정과 동일하게 **명시·결정적** 원칙. (필드 정의 → tech-stack §3.1.1)
- **SeriesCard**: cover(s.image) · title · description · **"N편"**(속한 글 수) · 최신 업데이트일.
- 각 그룹 내 정렬: **최신 활동순**(시리즈 내 가장 최근 글 날짜).
- 빈 그룹: 해당 섹션을 **헤더까지 통째로 숨김**(완결 시리즈 0개면 "완결" 섹션 자체가 안 뜸). 빈 시리즈(발행 글 0): 목록에서 숨김.

### 3.2 Series 상세 (`/series/[slug]`) — 순서 목록
- 헤더: series title · description · cover(있으면 히어로)
- **`order` 오름차순** 글 목록(줄기-잎). 각 항목에 "1/N" 회차 표시.
- 셀렉터: `getPostsInSeries(slug)` (order 정렬), `listSeries()`.

## 4. About (`/about`) — 자기소개

### 구성 (준비되는 대로 채움)
- **프로필 헤더**: 사진 + 이름 + 한 줄 소개(Pretendard 부제, stone)
- **이력(career)**: 타임라인 형태
- **포트폴리오**: 프로젝트 카드(제목·설명·링크)
- **연락처/링크**: GitHub·메일·RSS

### 소스 정책
- **MDX 단일 페이지(`content/about.mdx`)** 로 작성 추천 — 글처럼 편집 가능하고, 커스텀 컴포넌트(`ProfileHeader`·`Timeline`·`ProjectCard`)를 MDX 맵으로 주입. (대안: 코드 전용 뷰 — 편집 유연성↓)
- 레이아웃: 읽기 단일 컬럼 재사용. 사진·프로젝트 이미지는 "글에 집중" 원칙상 **콘텐츠에 복무하므로 환영**.

## 5. 조회수 시스템 & 폴백 정책 (블로그는 절대 안 고장)

> `tech-stack.md §8`의 애널리틱스를 **"보류 → 채택(조회수 한정)"** 으로 승격. 저장소는 **Upstash Redis(Vercel Marketplace 통합)**, **무료 플랜 · auto-upgrade OFF**(초과 시 과금 아니라 차단 → 과금 리스크 0).

### 5.1 플로우
| 단계 | 방법 |
|---|---|
| 저장소 | Upstash Redis (Vercel Marketplace, env 자동 주입) |
| 집계(쓰기) | 글 상세에서 클라이언트가 route handler 호출 → 중복방지 `SET dedup:{ipHash}:{slug} NX EX 86400` → `ZINCRBY views {slug} 1` |
| 개별 표시(선택) | `ZSCORE views {slug}` |
| 랭킹(읽기) | Home ISR → `ZREVRANGE views 0 {N-1} REV` → slug∩Velite → 메타 매핑 |

### 5.2 폴백 정책 — 표면별로 못박음

**대원칙:** 조회수는 **비필수 향상 기능**이다. 모든 Redis 접근은 격리(try/catch + 타임아웃)하고, 실패하면 **빌드타임에 이미 아는 Velite 데이터로 우아하게 대체**한다. 렌더링은 Redis 성공에 절대 의존하지 않는다.

| 표면 | 정상 | 실패(타임아웃·차단·미설정·에러) 시 |
|---|---|---|
| **집계(쓰기, 클라이언트)** | fire-and-forget, 2s 타임아웃, 24h 중복방지 | **조용히 무시**(swallow). UI·네비게이션 영향 0. 카운트만 안 오름 |
| **Home "가장 많이 본"(ISR 읽기)** | top-N ∩ Velite | ① 재검증 중 실패 → Next가 **직전 성공 정적본 유지**(stale-while-revalidate), 에러 안 뜸 · ② 최초/완전 실패 → **"가장 많이 본" 섹션만 통째로 생략**(바로 아래 "최근 글" 섹션이 이미 있으므로 화면은 멀쩡). 가짜 "최신" 재라벨 안 함 |
| **Home "최근 글"** | Velite 날짜 정렬 | Redis 무관 → **항상 표시**(Home의 정적 안전판) |
| **개별 글 조회수(선택 노출)** | `ZSCORE` 숫자 | 숫자 **완전히 숨김** (0 표기·에러 표기 안 함) |
| **데이터 정합** | Redis slug ∩ Velite 현재 글 | 삭제된 글 slug는 랭킹에서 제외(map miss 필터) |

### 5.3 추가 안전장치
- **env 가드:** Redis 호출 전 env 존재 확인 → 없으면 즉시 폴백. 로컬·미설정 환경에서도 빌드·렌더 정상.
- **타임아웃:** `AbortController` 상한(쓰기 2s / 읽기 1.5s). 초과 시 폴백.
- **봇/prefetch 무시:** UA·`Sec-Purpose: prefetch` 헤더면 카운트 안 함(정합성 보호).
- **절대 규칙:** **어떤 Redis 실패도 `throw`로 전파해 페이지를 깨뜨리지 않는다.**
  - ↔ 대조: Velite **빌드 검증**은 일부러 깨뜨린다(tech-stack §3.1.1). 그건 빌드타임 데이터 정합, 이건 런타임 비필수 — **정책이 정반대**임에 주의.

## 7. 빈 상태 · 에러 상태 문구

### 7.0 공통 패턴
모든 빈/에러 상태는 하나의 조용한 컴포넌트로 통일 — **잎 1개(`#leaf` outline, muted) + eyebrow(Pretendard·moss·500) + 한 줄 메시지 + 액션 링크 1개.** 스피너·애니메이션·삽화 없음(원칙: 산만함 배제). 404/500은 **명확성 우선**, 시적 표현은 보조. `role="status"`(빈) / `role="alert"`(에러) + 포커스 이동.

### 7.1 화면별 문구

| 표면 | 조건 | 동작·문구 |
|---|---|---|
| **Home** | 글 0편(첫 실행) | eyebrow *dongCoding* · "아직 심어둔 글이 없습니다. 곧 첫 잎이 돋아납니다." · (액션: "소개 보기 →" `/about`) |
| **Home — 가장 많이 본** | 조회수 실패 | 섹션 **통째로 생략**(§5.2). 최근 글이 화면을 지킴 |
| **Posts** | 발행 글 0편 | eyebrow *Empty* · "아직 글이 없습니다." · (액션: "dongCoding 소개 →" `/about`) |

| **About** | 콘텐츠 미준비 | 최소 플레이스홀더: "소개가 곧 준비됩니다." (about.mdx는 커밋 콘텐츠) |
