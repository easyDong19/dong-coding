# dongCoding — 테스트 기획 (Test Plan)

> 이 문서는 **"어디에 테스트를 써야 하는가"** 를 확정한 기획이다.
> 목표는 커버리지 숫자가 아니라 **① 순수 함수로 격리된 로직, ② 실제로 존재하는 엣지 케이스** 두 가지다.
> 데이터 계약은 `content-plan.md`, 정책 근거는 `pages-plan.md`, 아키텍처는 `tech-stack.md`를 따른다.

---

## 0. 테스트 원칙 (무엇을 테스트하고, 무엇을 안 하나)

### 0.1 테스트한다 — 값이 나오는 곳

1. **순수 함수만 유닛 테스트 1급 대상.** `(입력) → 출력`이 결정적인 셀렉터·계산 로직. 목(mock) 없이 픽스처 배열만 넣으면 검증되는 코드.
2. **엣지 케이스가 정책으로 존재하는 곳.** 기획 문서에 "첫 회차면 이전 빈칸", "0편이면 섹션 생략", "범위 밖이면 404"처럼 **분기가 명문화된 지점**이 테스트 대상이다. 테스트는 이 정책의 실행 가능한 사본이 된다.
3. **실패가 조용해야 하는 곳.** 조회수 폴백(`pages-plan §5.2`)처럼 "실패해도 안 깨진다"가 요구사항인 로직은, 실패 경로가 눈에 안 보이므로 **테스트 없이는 검증 자체가 불가능**하다.

### 0.2 테스트하지 않는다 — 커버리지용 테스트 금지

| 안 하는 것 | 이유 |
|---|---|
| Velite 내장 기능 (`s.toc()`, `s.image()`, `s.excerpt()`, `s.metadata()`) | 외부 라이브러리의 책임. 우리가 검증할 계약이 아님 |
| rehype-pretty-code · KaTeX 렌더 결과 | 빌드타임 외부 파이프라인. 스냅샷 테스트는 유지비만 발생 |
| 정적 UI 컴포넌트의 마크업/스타일 | 토큰·시각 검증은 프로토타입/눈으로 (design.md가 SSOT) |
| Next.js 프레임워크 동작 (ISR 타이머, `notFound()` 라우팅, `generateStaticParams` 자체) | 프레임워크의 책임. 우리는 **넘겨주는 값**만 테스트 |
| 단순 위임 컴포넌트 (props 그대로 뿌리는 것) | 분기 없는 코드에 테스트는 소음 |

> **판단 기준 한 줄:** *"이 테스트가 깨지면 내 로직의 버그인가, 남의 라이브러리 버전업인가?"* — 후자면 쓰지 않는다.

### 0.3 테스트 가능하게 만드는 구조 규칙 (아키텍처 계약)

- 모든 계산 로직은 `entities/{post,series}/model/selectors.ts`에 **순수 함수**로 둔다. `.velite` import는 진입 파일에서만 하고, **셀렉터 자체는 `posts: Post[]`를 인자로 받는 형태**로 작성한다 → 테스트에서 픽스처 배열 주입 가능, 목 불필요.
- 조회수 관련 판정(봇 감지, dedup 키 생성)은 Redis 클라이언트와 **분리된 순수 함수**로 뽑는다. Redis I/O는 얇은 껍데기로 남긴다.
- Velite `prepare` 훅의 검증 로직은 인라인으로 쓰지 말고 **named 함수로 분리**해 config 밖에서 import → 테스트 가능.

### 0.4 도구

- **Vitest** — Next 16 + TypeScript + ESM 정합, 설정 최소. `pnpm test` 스크립트 추가.
- 픽스처: `Post`/`Series` 타입의 최소 객체를 만드는 팩토리 헬퍼 (`makePost({series: 'x', order: 1})`) 하나면 충분.
- E2E·비주얼 테스트는 **도입 안 함**(YAGNI — 1인 블로그. 320px 무깨짐 검증은 프로토타입 단계 수동 확인).

## 1. P0 — 시리즈 셀렉터 (엣지 케이스 밀도 최고)

시리즈는 이 블로그에서 관계 계산이 가장 많은 1급 엔티티(`tech-stack §3.1.1`). 순서·이웃·경계 로직이 전부 여기 모인다.

### 1.1 `getSeriesNav(postSlug)` → `{series, index, total, prev, next}`

| # | 케이스 | 기대 동작 (근거) |
|---|---|---|
| 1 | 중간 회차 (2/5) | `prev`·`next` 모두 존재, `index=2, total=5` |
| 2 | **첫 회차** | `prev = null` — 이전 자리 빈칸 유지 (design.md §4.6) |
| 3 | **마지막 회차** | `next = null` — 다음 자리 빈칸 유지 |
| 4 | **1편짜리 시리즈** | `prev`·`next` 모두 null, `total=1` — 2·3의 교집합 |
| 5 | **시리즈 미소속 글** | `null` 반환 → 컴포넌트 미표시 (pages-plan §7.1) |
| 6 | **order 빈틈** (1, 2, 5) | nav는 **order 정렬 기준 이웃**으로 동작 — 2의 next는 5. **`index`(N/M의 N)는 order 값이 아니라 정렬 위치**(1,2,5 중 세 번째면 `3 / 3`) — 확정 2026-07-02 |
| 7 | 시리즈 내 **draft 회차 존재** | draft는 빌드 제외 → nav 계산에서도 제외 (1,2(draft),3이면 1의 next는 3, `total`도 draft 뺀 수) |
| 8 | 존재하지 않는 slug | `null` (throw 금지 — 렌더 경로) |

> **확정(2026-07-02):** "N / M"의 N은 **정렬 배열에서의 위치**다. order 빈틈이 있어도 N ≤ M이 보장되고 `5 / 3` 같은 표시가 나오지 않는다. `order` 값은 이웃 계산(정렬·prev/next)에만 쓰고 표시엔 쓰지 않는다.

### 1.2 `getPostsInSeries(slug)` → order 오름차순 목록

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 정상 (order 3,1,2 입력) | 1,2,3 정렬 반환 — **입력 배열 순서에 의존하지 않음** |
| 2 | draft 포함 | 제외 후 반환 |
| 3 | 발행 글 0편 시리즈 | `[]` → 상세는 `notFound()` (pages-plan §7.1) |
| 4 | 없는 slug | `[]` |

### 1.3 시리즈 목록 그룹핑·정렬 (`/series`)

| # | 케이스 | 기대 동작 (pages-plan §3.1) |
|---|---|---|
| 1 | `complete: true/false` 혼재 | 진행 중 / 완결 두 그룹으로 분리 |
| 2 | 그룹 내 정렬 | **최신 활동순** = 시리즈 내 가장 최근 글 date의 내림차순 |
| 3 | **발행 글 0편 시리즈** | 목록에서 숨김 (최신 활동일 계산 불가 케이스이기도 함) |
| 4 | 완결 시리즈 0개 | 완결 그룹 `[]` → 뷰에서 섹션 통째 숨김 판정 가능해야 |
| 5 | "N편" 카운트 | draft 제외한 수 |

### 1.4 Velite `prepare` 훅 검증 함수 (빌드 게이트)

배포 차단 메커니즘 그 자체이므로 **throw 여부가 곧 스펙** (tech-stack §3.1.1).

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 없는 시리즈 참조 (`series: 'ghost'`) | **throw** (빌드 중단) |
| 2 | 같은 시리즈에 중복 order (1,1) | **throw** |
| 3 | order 빈틈 (1,3) | **throw 하지 않음** — 경고만 |
| 4 | `series` 있는데 `order` 없음 | refine 실패 (스키마 층) |
| 5 | 정상 데이터 | 통과 |
| 6 | **draft 글의 잘못된 참조** | **throw 하지 않음** — draft는 검증 대상에서 제외 (확정 2026-07-02). draft는 빌드 제외라 프로덕션 무영향이고, 초안이 배포를 막지 않아 집필 흐름 보존 |

> **확정(2026-07-02):** prepare 훅의 참조 무결성·order 중복 검증은 **발행 글(draft:false)만** 대상으로 한다. 검증 함수는 `posts.filter(p => !p.draft)`를 먼저 적용한 뒤 무결성을 확인한다 — 테스트도 "draft 오류는 통과, 발행 글 오류는 throw" 쌍으로 고정.

## 2. P0 — 페이지네이션 계산 (`/posts/page/[n]`)

경계값(off-by-one)의 고전적 서식지. 페이지 크기 10 (pages-plan §2).

### 2.1 페이지 분할 `paginate(posts, page, size)`

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 25편, page 2 | 11~20번째 글, `totalPages=3` |
| 2 | **정확히 10편** (경계) | `totalPages=1` — 11이 아님 |
| 3 | **11편** (경계+1) | `totalPages=2`, 2페이지엔 1편 |
| 4 | **0편** | `totalPages=0` 또는 1 — 빈 상태 렌더와의 계약 확정 필요, `/posts/page/2`는 404 |
| 5 | **범위 밖** (`page=999`) | `notFound()` 신호 (pages-plan §7.1) |
| 6 | `page=0`·음수·비정수 | `notFound()` 신호 |
| 7 | `page=1` 직접 접근 | `/posts` 영구 리다이렉트 신호 (정규 URL 정책) |
| 8 | draft 혼재 | 제외 **후** 페이지 계산 (제외 전 계산이면 마지막 페이지가 빌 수 있음 — 흔한 버그) |

### 2.2 페이저 표시 윈도 `getPageItems(current, total)` → `‹ 1 2 3 … M ›`

`…` 생략 계산은 분기가 많아 순수 함수 분리가 필수.

| # | 케이스 | 기대 출력 예 |
|---|---|---|
| 1 | total 적음 (≤5 등) | 생략 없이 전부: `1 2 3` |
| 2 | current가 앞쪽 (1/10) | `1 2 3 … 10` |
| 3 | current가 중간 (5/10) | `1 … 4 5 6 … 10` |
| 4 | current가 끝 (10/10) | `1 … 8 9 10` |
| 5 | **생략이 1페이지만 가리는 경우** | `1 … 3`이 아니라 `1 2 3` — `…`가 숫자 하나보다 길면 무의미 |
| 6 | total=1 | 페이저 자체 미표시 판정 |

## 3. P1 — 목록·홈 셀렉터

### 3.1 글 정렬·필터 `getPublishedPosts()`

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 날짜 내림차순 | 최신 우선 |
| 2 | draft 제외 | 프로덕션 목록·RSS·시리즈·페이지네이션 **전부의 상류**이므로 여기 한 곳에서 보장 |
| 3 | **같은 날짜 2편** | **slug 사전순** tie-break (확정 2026-07-02) — 결정적이라 빌드마다 순서 불변 → ISR/스냅샷 안정. `date desc → slug asc` 복합 정렬을 테스트로 고정 |
| 4 | **미래 날짜 글** | **그냥 노출** (확정 2026-07-02) — 예약 발행 개념 없음. date 필터링 없이 정렬만. 이 동작을 테스트로 명시 고정(추후 예약 발행 도입 시 회귀 방지) |

### 3.2 조회수 랭킹 교집합 `rankToPosts(slugsFromRedis, posts)` (pages-plan §5.2)

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 정상 top-5 | Redis 순서 유지 + Velite 메타 병합 |
| 2 | **삭제된 글 slug 포함** | 필터 후 반환 (map miss 필터 — 명문화된 정합 규칙) |
| 3 | **삭제 필터 후 N 미달** (5 요청, 3 유효) | **모자라는 대로 3개 반환** (확정 2026-07-02) — `ZREVRANGE`는 정확히 N만 읽고, 필터 후 남는 만큼만. 다음 순위로 채우지 않음(단순·정직). 삭제 글은 드물어 실질 영향 작음 |
| 4 | Redis 빈 배열 (조회 0) | `[]` → "가장 많이 본" 섹션 통째 생략 판정 |
| 5 | draft/비공개 slug가 Redis에 남은 경우 | 교집합에서 자연 배제되는지 |

### 3.3 관련 포스팅 `getRelatedPosts(post, posts)` — 태그 기반

| # | 케이스 | 기대 동작 |
|---|---|---|
| 1 | 공유 태그 다수 | 겹치는 태그 수 내림차순 (동점 시 최신순 — 결정적 tie-break) |
| 2 | **자기 자신** | 반드시 제외 (누락하기 가장 쉬운 버그) |
| 3 | 매칭 0편 | `[]` → 섹션 통째 생략 (pages-plan §7.1) |
| 4 | 대상 글에 태그 없음 | `[]` (에러 아님) |
| 5 | **같은 시리즈 글의 취급** | **제외하지 않음 — 태그만 겹치면 포함** (확정 2026-07-02). series-nav와 중복될 수 있으나, 관련도는 태그 기준으로만 단순 계산. 시리즈 여부로 분기하지 않음 → 셀렉터가 시리즈 정보에 의존하지 않아 순수성·테스트 단순성 유지 |

## 4. P1 — 조회수 판정·폴백 (순수 판정 + 격리 실패)

"블로그는 절대 안 고장"(pages-plan §5)의 실행 검증. **I/O는 목 없이 테스트할 수 없으므로, 판정을 순수 함수로 뽑는 것이 전제**(§0.3).

### 4.1 순수 판정 함수

| 함수 | 케이스 |
|---|---|
| `isBotOrPrefetch(headers)` | 알려진 봇 UA → true · `Sec-Purpose: prefetch` → true · 일반 브라우저 → false · **헤더 자체 부재** → false (카운트 허용) |
| `dedupKey(ipHash, slug)` | 형식 `dedup:{ipHash}:{slug}` 고정 — 키 형식 변경은 곧 중복방지 리셋이므로 테스트로 동결 |
| `hashIp(ip)` | 같은 IP → 같은 해시(결정적) · 다른 IP → 다른 해시 · **원문 IP가 출력에 포함되지 않음** (프라이버시) |

### 4.2 폴백 경로 (Redis 클라이언트만 목)

| # | 케이스 | 기대 동작 (pages-plan §5.2·§5.3) |
|---|---|---|
| 1 | **env 미설정** | Redis 호출 자체를 안 함 + 즉시 폴백값 — 로컬 빌드가 그냥 됨 |
| 2 | 타임아웃 (읽기 1.5s 초과) | 폴백값 반환, **throw 전파 없음** |
| 3 | Redis 예외 | 동일 — 어떤 실패도 렌더를 못 깨뜨림 (절대 규칙) |
| 4 | 쓰기 실패 | 조용히 무시(swallow) — 반환값·에러 없음 |
| 5 | `ZSCORE` 실패 시 개별 조회수 | `null` 반환 → 뷰에서 숫자 숨김 (0 표기 금지) |

## 6. 우선순위 요약

| 순위 | 대상 | 이유 |
|---|---|---|
| **P0** | 시리즈 셀렉터 (§1) | 엣지 케이스 밀도 최고 + prepare 훅은 배포 게이트 그 자체 |
| **P0** | 페이지네이션 (§2) | off-by-one 서식지, 404/리다이렉트 정책 분기 |
| **P1** | 목록·홈 셀렉터 (§3) | 모든 화면의 상류 데이터, 교집합·tie-break 결정성 |
| **P1** | 조회수 판정·폴백 (§4) | "실패해도 안 깨짐"은 테스트 없인 검증 불가 |
| **P2** | 피드·테마 (§5) | 분기는 있으나 규모 작고 실패 파급 작음 |

## 7. 테스트가 드러낸 스펙 (2026-07-02 전량 확정)

이 문서를 쓰면서 발견된, 기획 문서에 답이 없던 분기 7건. **테스트를 쓰려다 드러난 기획 구멍**이며, 아래처럼 전부 확정해 각 케이스 표에 반영했다.

| # | 위치 | 질문 | 확정 |
|---|---|---|---|
| 1 | §1.1-6 | order 빈틈 시 "N / M"의 N | **정렬 위치** (order 값 아님) — N ≤ M 보장 |
| 2 | §1.4-6 | draft 참조 오류도 빌드 중단? | **아니오** — prepare 검증은 발행 글만 대상 |
| 3 | §3.1-3 | 같은 날짜 tie-break | **slug 사전순** (`date desc → slug asc`) |
| 4 | §3.1-4 | 미래 날짜 글 노출 | **그냥 노출** (예약 발행 없음) |
| 5 | §3.2-3 | 랭킹 N 미달 시 채움? | **아니오** — 모자라는 대로 반환 |
| 6 | §3.3-5 | 관련 글에서 같은 시리즈 취급 | **제외 안 함** — 태그만 겹치면 포함 |
